Case study 01 · Design systems

Multi-brand design system: one source of truth for three brands

Role Design system owner
Scope Full system — tokens, components, templates
Consumers Designers, developers, marketing
Tools Figma variables & modes, Storybook
Case 01

Context

The company operates three separate brands on a single codebase — Brand A as the main product, Brand B as a localised variant serving the Netherlands market, and Brand C as a distinct brand with a different visual identity. Each brand has its own colour scheme, typography, and in some cases different form requirements driven by local regulations.

Before this project, each brand had its own Figma file maintained separately. Any screen shared across brands — registration flows, doctor consultation pages, account pages — had to be manually replicated. Changing a heading style in one file meant repeating the same operation in the other two, property by property: font family, weight, line height, letter spacing.

That fragmentation was costing the team significant time and introducing inconsistencies. The goal was to consolidate everything into a single design system serving all three brands from one source, without sacrificing brand differentiation.

Case 01

Problem

Two compounding problems

The challenge wasn't just cosmetic. The three brands had genuinely different design languages that couldn't simply share a stylesheet.

  • Brand A and Brand B both use Roboto. Brand C uses Satoshi — a font with different optical metrics, different weight behaviour, different line height needs. The same "small heading" style required completely different values depending on the brand.
  • Some components needed to show different fields per brand. The Netherlands market (Brand B) required a BSN identification number and a medication data sharing consent block. Hard to maintain across three separate files.
  • The header needed a different logo per brand. A single flexible component had to accommodate all three without duplicating the structure.

The typography challenge

The difference between Roboto and Satoshi goes beyond swapping a font family. A heading set in Roboto Bold at 26px/36px needs to become Satoshi Medium at 24px/29px to achieve the same visual presence — and that shift needs to be automatic, not something a designer has to remember each time.

The same heading component in Brand A (Roboto, 26px/700/36px) and Brand C (Satoshi, 24px/500/29px). Swapping only the font family breaks the layout — all metrics need to change simultaneously.

Case 01

Process

Building the token architecture

The solution required thinking in three levels of abstraction. The key insight was that the semantic layer is what changes per brand — the base palette and component structure remain identical across all three.

Token architecture — a single mode switch propagates through all three levels. Only the semantic layer differs per brand; base and component structure are shared.

1

Base tokens

A complete palette of colour ramps — Grey, Blue, Green, Orange, Red, Yellow — each from stop 50 to 950. Every brand shares the same base set.

2

Semantic tokens

brand-default border maps to Orange/500 in Brand A, Green/500 in Brand B, Blue/500 in Brand C. Typography tokens are also defined here — font family, weight, line height, letter spacing — so switching brand mode changes all metrics at once.

3

Component tokens

Per-state variables for every component — default, hovered, pressed, disabled. These reference the semantic layer, so a primary button automatically uses whichever accent colour the active brand defines.

Typography encoded into modes

Each text style carries not just font family but weight, line height, and letter spacing as mode-aware variables. Switching from Brand A to Brand C changes all of these simultaneously with no manual intervention.

The primary button across all three brand modes — one component definition, five states each, three completely different visual outputs driven entirely by the token architecture.

Component flexibility for form fields

The shipment form needed different fields per brand depending on local regulations. Boolean properties and mode-aware visibility let one component cover all three cases without maintaining separate versions.

The delivery details form across all three brands. Brand B shows additional fields required by Dutch regulations — BSN number and medication data consent. One component, three configurations.

Case 01

Solution

One system, one toggle

A single Figma design system where switching a mode variable transforms every aspect of the brand simultaneously — colour scheme, full typography stack, component field visibility, logo, and accent colour. The same switch is mirrored in Storybook using identical semantic variable names, so designers and developers always work from the same source of truth.

The system is consumed by three groups: product designers in their daily work, developers who base their UI library on the same semantic variables, and the graphic design team who use the semantic tokens to align marketing materials with the product.

A cross-section of components in all three brand modes. Subscription cards, prescription alert dialogs, input fields, and order summaries — all driven by a single component set and the token architecture.

Case 01

Outcome

A full day of work became ten minutes

Before the consolidated system, updating a shared screen across all three brands meant manually replicating every property in three separate files. With the mode-based system, the same change takes a single toggle.

3→1
Figma files consolidated into a single source of truth
~10×
Faster brand-switching for day-to-day design work
3+
Consumer groups aligned — design, engineering, marketing

Scalability as validation

The most meaningful indicator of a well-architected system isn't that it solved the problems you designed it for — it's that it handles problems you didn't anticipate. The mode-based approach has been applied to new regional requirements, new component variants, and cross-team marketing assets without structural changes to the system.

"There was no before/after to show — the before state was fragmentation, not a different visual design. The value of the system isn't visible in a single screenshot; it's in the work that no longer needs to happen."

Case study 02 · Healthcare product design

Weight management: scaling a subscription model across three medications

Role Sole product designer
Timeline ~2 months
Domain Healthtech / e-commerce
Support Head of Product, Product Owner
Case 02

Context

Weight management was a new product category for the company — a prescription pharmacy platform. Three medications needed to be introduced to users: Wegovy, Saxenda, and Mounjaro. All three fall under the same category, but each has a fundamentally different treatment model, injection type, and dosage logic. The task was to design a single, coherent purchase and subscription flow that worked for all three.

I was the sole designer on this project, conducting most of the research and owning the UX end to end, supported by the Head of Product and a responsible PO for business model questions.

Overview of the three weight management medications and their key differences. Each has a distinct treatment scheme — a single unified UI had to accommodate all three without confusing users.

Case 02

Problem

Three medications, three different treatment logics

The complexity wasn't just about showing different products — it was that each product behaved differently over time and couldn't simply reuse the same UI patterns.

  • Wegovy uses a dose escalation scheme: the dosage increases every four weeks according to a fixed clinical schedule. Users must receive the right dose at the right stage of treatment.
  • Saxenda contains a fixed daily injection amount. The key variable is volume — how many injections to include per shipment, calculated from the dosage and frequency.
  • Mounjaro uses a single pen with a large amount of medication that the user divides themselves, increasing the dose from the same container. This creates a different pack size and subscription logic entirely.

Dosage progression over time for all three medications. The different escalation patterns, durations, and maintenance doses meant the subscription model had to be designed differently for each — while appearing consistent to the user.

The subscription model didn't fit

The company's existing subscription model was built for simple monthly replenishment: choose a dosage, choose a quantity, receive the same thing every month. None of the three weight management medications worked that way. Dosages change over time. The clinical 4-week cycle doesn't align with calendar months. And the right next dosage for a user depends on how their treatment is progressing — information gathered during an earlier consultation.

Six different user types with conflicting needs

Working from a list of personas provided by the product team, I mapped out the pain points, jobs to be done, and likely behaviour patterns for six distinct user types — from first-time patients starting treatment to users who had already begun elsewhere and were switching pharmacies mid-journey.

Six user types identified for the weight management category. Each had different starting points, different levels of medical knowledge, and different expectations of the purchase flow.

User stories mapped per persona, used to validate that the designed flow covered the needs of all six user types — including edge cases like mid-journey switchers and users experiencing side effects.

Case 02

Process

Mapping the logic before designing

Before wireframing anything, I built out the decision logic that would drive what dosages to show, in what order, and with what default selected — based on answers a user had given during the medical consultation earlier in the checkout flow. This was the foundation that everything else was built on.

The full dosage logic flow — branching by medication, prior usage status, current dosage, and side effect history. This document was used to align with the product team and developers before any UI was built.

1

Contextual dosage display

Only medically valid next dosages are shown, based on the user's consultation answers. A first-time user sees a different set of options than someone already mid-treatment — filtering out clinically impossible choices reduces cognitive load and prescription risk.

2

Subscription with dosage progression

Since dosage changes every four weeks, users need to understand that what they subscribe to today may change at the next cycle. A tooltip-based communication layer was designed to set this expectation without cluttering the purchase UI.

3

Persona validation

Each of the six personas was walked through the designed flow to verify their needs were met. This surfaced two edge cases that required changes to the default selection logic before handoff.

Iterating on dosage and pack size selection

The dosage selector went through several rounds of iteration. The central challenge was presenting options that looked consistent across three medications with very different pack structures — some with multiple pack sizes, some with none — without making the differences feel confusing or arbitrary to the user.

Iteration history for the Mounjaro dosage selector. Mounjaro's single-pen model — where the user doses from the same container rather than choosing between separate pens — required a different layout approach compared to Wegovy and Saxenda.

Case 02

Solution

A unified UI that adapts per medication

The final design is a single dosage and subscription selector that adapts its structure, options, and communication layer to the medication type — without exposing that complexity to the user. The layout is consistent enough to feel familiar, and different enough to be accurate for each product's clinical reality.

Due to technical resource constraints at the time of development, the implemented version differs slightly from the final designs — some of the more nuanced UI details were descoped. The screens below reflect what was built and is live in production.

The final production UI across all three weight management medications. Wegovy shows pen-based dosage options with a subscription plan selector. Mounjaro shows the KwikPen format with a single dosage option. Saxenda shows volume-based options across different pack sizes.

Case 02

Outcome

A new product category, successfully launched

All three medications were integrated into the platform and are actively purchased by users. The flow handles six distinct user personas, three different treatment models, and a subscription logic that didn't previously exist in the company's product.

3
medications launched under one unified flow
6
user personas validated against the design
~2mo
from first brief to shipped product, sole designer

What I'd do differently

The subscription model for dosage-escalating medications is inherently complex, and the tooltip-based communication approach — while practical within the technical constraints — is a compromise. In an ideal scenario, the system would automatically propose the next dosage at the right moment, rather than relying on the user to remember and act. That remains an open design problem worth returning to.

Subscription cancellation rates across the platform dropped from 40% to 14% following a separate project to redesign the subscription management flow — a strong signal that users are engaging with the service long-term once they're past the initial purchase.

"The hardest part wasn't designing the UI — it was understanding the clinical reality well enough to design something that was both accurate and usable. The decision logic diagram turned out to be as important an artifact as any screen."
